Problem Context 01

Lead sequences live across channels. Email, SMS, calls, and manual tasks execute independently without unified visibility.

Status lives in dashboards. Teams refresh reports to understand what needs attention, who's engaged, and which sequences stall.

Most tools optimize for execution, not orchestration or insight.

What if lead management worked like a state machine?

  • System tracks every touchpoint across channels with explicit state transitions.
  • Users see real-time status without polling APIs or refreshing dashboards.
  • Sequence logic is explicit, auditable, and centrally orchestrated.

Not a CRM clone. Not a notification system. A sequence orchestration and visibility layer.

System Architecture

Event-driven orchestration core. Centralized state machine manages sequence progression, not scattered webhook handlers.

Domain-driven design. Models explicitly represent Sequences, Leads, Steps, Events, and Constraints.

Channel adapters as data sources. Email, SMS, and call providers are passive event sources. Logic lives in orchestration layer.

Architecture is product.

Stack & Flow Frontend ‚Üí Backend ‚Üí Providers
Frontend Dashboard

Real-time WebSocket updates, no polling. Focused on visibility and action, not data entry.

Orchestration Service

Owns sequence state, event processing, conditional logic, and metric aggregation.

Channel Adapters

Normalized interfaces to email, SMS, call tracking. Return events only.

Event Processing Lifecycle

01 ¬∑ Event Capture

Email opened, SMS replied, call completed, or manual task logged.

02 ¬∑ Validation & Enrichment

Verify event authenticity, normalize schema, attach lead and sequence context.

03 ¬∑ State Transition

Orchestration layer evaluates rules, updates sequence state, determines next step.

04 ¬∑ Action Dispatch

If conditions met, trigger next channel action or flag for manual intervention.

05 ¬∑ Metric Update

Aggregate engagement data, goal progress, and sequence health metrics.

06 ¬∑ Dashboard Sync

Push real-time updates to frontend via WebSocket. No refresh needed.

Design Principles

Architecture is product. System design directly affects reliability, debuggability, and team velocity.

Visibility over automation. Show state explicitly. Make transitions auditable and reversible.

Centralized orchestration. One service owns sequence logic. Providers are passive data sources.

This is not a notification engine. It is a system for understanding, orchestrating, and monitoring lead sequences.
